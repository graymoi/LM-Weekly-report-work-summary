# 验证循环 Hook

## 功能说明

在关键操作前后执行验证，确保系统行为的正确性和质量。

## 触发时机

- Git提交前（pre-commit）
- 文件保存后
- 任务完成后
- 用户手动触发

## 验证类型

### 1. 文件命名验证

验证文件命名是否符合规范：

| 文件类型 | 命名规则 | 示例 |
|---------|---------|------|
| 日总结 | `YYYY-MM-DD-summary.md` | `2026-03-14-summary.md` |
| 周总结 | `MM.DD-MM.DD周工作总结.md` | `3.10-3.14周工作总结.md` |
| 政策文件 | `[发文号]_[发文机关]_[政策名称]_[发布日期].md` | `国办发〔2024〕52号_国务院办公厅_关于优化完善地方政府专项债券管理机制的意见_20241225.md` |
| 对话日志 | `对话日志_YYYY-MM-DD.md` | `对话日志_2026-03-14.md` |

### 2. 敏感数据检查

检查是否包含敏感信息：

| 检查项 | 正则表达式 | 处理方式 |
|-------|-----------|---------|
| API密钥 | `[a-zA-Z0-9]{32,}` | 警告并阻止 |
| 密码 | `password\s*[:=]\s*['\"]?[^'\"\s]+` | 警告并阻止 |
| 手机号 | `1[3-9]\d{9}` | 提示脱敏 |
| 身份证 | `\d{17}[\dXx]` | 提示脱敏 |

### 3. 内容完整性验证

验证文件内容是否完整：

| 检查项 | 说明 | 处理方式 |
|-------|------|---------|
| 必填字段 | 检查元数据必填字段 | 警告 |
| 空文件 | 检查文件是否为空 | 警告 |
| 格式正确 | 检查Markdown格式 | 提示 |

### 4. 逻辑一致性验证

验证逻辑是否一致：

| 检查项 | 说明 | 处理方式 |
|-------|------|---------|
| 日期逻辑 | 检查日期是否合理 | 警告 |
| 引用检查 | 检查引用文件是否存在 | 提示 |
| 状态一致 | 检查状态字段是否一致 | 警告 |

---

## 验证流程

```
触发验证
    │
    ├── 文件命名验证
    │       ├── 通过 → 继续
    │       └── 失败 → 提示修正
    │
    ├── 敏感数据检查
    │       ├── 通过 → 继续
    │       └── 发现敏感数据 → 阻止并警告
    │
    ├── 内容完整性验证
    │       ├── 通过 → 继续
    │       └── 不完整 → 提示补充
    │
    └── 逻辑一致性验证
            ├── 通过 → 完成
            └── 不一致 → 提示修正
```

## Checkpoint验证

### 检查点类型

| 检查点 | 触发时机 | 验证内容 |
|-------|---------|---------|
| 任务开始前 | 任务初始化 | 前置条件检查 |
| 任务进行中 | 关键步骤 | 中间状态验证 |
| 任务完成后 | 任务结束 | 结果验证 |

### Pass@K机制

```
Pass@K: 在K次尝试中至少成功一次的概率

计算公式:
Pass@K = 1 - (n-k choose K) / (n choose K)

其中:
- n: 总尝试次数
- k: 成功次数
- K: 设定的尝试次数

示例:
- Pass@1: 第一次就成功的概率
- Pass@3: 三次尝试中至少成功一次的概率
```

---

## 配置选项

在 `hook-config.yaml` 中配置：

```yaml
verification:
  enabled: true
  checkpoint_enabled: true
  continuous_eval: false
  pass_at_k: 3

pre_commit:
  enabled: true
  validate_file_naming: true
  check_sensitive_data: true
```

## 最佳实践

### 1. 分级验证

| 级别 | 验证内容 | 失败处理 |
|------|---------|---------|
| 严重 | 敏感数据、安全风险 | 阻止操作 |
| 警告 | 命名规范、格式问题 | 提示但允许继续 |
| 提示 | 优化建议、最佳实践 | 仅提示 |

### 2. 快速失败

- 发现严重问题立即停止
- 避免浪费时间在错误的基础上
- 提供清晰的错误信息

### 3. 持续改进

- 收集验证失败数据
- 分析失败原因
- 优化验证规则
